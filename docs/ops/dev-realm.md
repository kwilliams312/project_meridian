# Dev realm (`meridian-dev`) ‚Äî GitOps runbook

The **Dev** hosted realm (D-37 / SYNC ¬ß16). ArgoCD keeps namespace `meridian-dev`
in sync with the `dev` branch: `authd` + `worldd` + an **ephemeral** in-cluster
MariaDB, self-signed TLS, exposed on fixed NodePorts. Phase 1 of the
[CD hosted-realms design](../superpowers/specs/2026-07-07-cd-hosted-realms-design.md).

## Endpoints

| Daemon | ClusterIP (in-cluster) | External (shared MetalLB LB IP) |
|--------|------------------------|---------------------------------|
| authd  | `meridian-dev-authd:7100`  | `192.168.89.31:7100` |
| worldd | `meridian-dev-worldd:7200` | `192.168.89.31:7200` |
| mariadb | `meridian-dev-mariadb:3306` | ‚Äî (in-cluster only) |

authd and worldd **share one MetalLB LoadBalancer IP** (`192.168.89.31`) via the
`metallb.universe.tf/allow-shared-ip` annotation, so a single realm IP reaches
both (authd :7100, worldd :7200). The in-cluster load harness uses the ClusterIP
names. ‚öÝÔ∏è authd does not yet advertise this IP in its realm list ‚Äî external
*real* clients need that (tracked in [#339](https://github.com/kwilliams312/project_meridian/issues/339)).

Both daemons speak **raw TLS 1.3** (not HTTP). TLS is a **per-pod self-signed
cert** generated by a `certgen` initContainer (CN = the daemon's service name) ‚Äî
clients must pin/skip-verify (dev only). PTR/Prod use real certs via
`tls.existingSecret`.

## Register / operate

```bash
# One-time: register the app-of-apps (creates + auto-syncs meridian-dev).
kubectl apply -f deploy/gitops/root-app.yaml

# Force an immediate reconcile (otherwise ArgoCD polls git ~every 3 min):
kubectl -n argocd annotate application meridian-dev argocd.argoproj.io/refresh=hard --overwrite

# Status:
kubectl -n argocd get application meridian-dev -o \
  jsonpath='{.status.sync.status} / {.status.health.status}{"\n"}'
kubectl -n meridian-dev get pods -o wide
```

Dev auto-syncs (prune + selfHeal) from `dev`; a push to `dev` rolls out. The DB
is ephemeral (`emptyDir`) ‚Äî it reseeds the 3-DB split on every mariadb pod start.

## Images

Built by [`.github/workflows/dev-images.yml`](../../.github/workflows/dev-images.yml)
on push to `dev` (amd64, `:dev` tag): `authd`, `worldd` (from source) and
`meridian-db` (self-seeding, also `:latest`). Pods are **pinned to amd64 nodes**
(temporary ‚Äî removed in Phase 2 when images go multi-arch).

## Server-authoritative characters (D-35 / #341)

worldd is the source of truth for characters. `WORLD_HELLO` lands the session at
character-select (no spawn); the client must send an explicit
`ENTER_WORLD(character_id)` and worldd spawns only if that character is **owned by
the session's account** (else a typed `NOT_FOUND`/`NO_CHARACTER`). The old
hardcoded "Placeholder" fabrication is gone, so **worldd must reach
`meridian_characters`** ‚Äî the chart wires all five `MERIDIAN_CHARDB_*` vars
(`worldd.chardb.enabled: true`, name `meridian_characters`). Verify on the realm:

```bash
kubectl -n meridian-dev exec deploy/meridian-dev-worldd -- \
  sh -c 'echo $MERIDIAN_CHARDB_HOST $MERIDIAN_CHARDB_USER $MERIDIAN_CHARDB_NAME'
# ‚Üí meridian-dev-mariadb meridian meridian_characters   (all non-empty)
```

If the chardb env is empty, every `ENTER_WORLD` returns `INTERNAL` and the realm
is unenterable. The concurrency harness pre-creates one character per account
(`add-characters` runs before the bots), so the bots enter as real, persisted
characters (`scripts/dev/loadtest.sh` step 2/3).
